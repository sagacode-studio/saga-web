Dokumen ini adalah instruksi wajib bagi @Programmer, @Designer, dan @Researcher untuk menghemat penggunaan token tanpa mengurangi kualitas produk.1. Prinsip Output "Kikir" (Minimalis Berdampak Tinggi)No Yapping: Jangan memberikan penjelasan panjang lebar atau basa-basi pembuka/penutup yang tidak perlu. Langsung ke inti jawaban.Dry Code: Tulis kode yang padat. Hindari komentar berlebihan di dalam file kodingan kecuali untuk logika yang sangat kompleks.Diffs Only: Jika melakukan perbaikan pada file yang sudah ada, jangan tulis ulang seluruh isi file. Berikan hanya bagian (fragmen) kode yang berubah atau perlu ditambah/dihapus.2. Struktur Komunikasi InternalInternal Brief: Saat berdiskusi antar sub-agen, gunakan poin-poin (bullet points) singkat. Jangan gunakan kalimat naratif panjang.Chain of Thought Ringkas: Jika harus "berpikir" (thinking process), rangkum dalam satu kalimat teknis saja.3. Aturan Teknis Penulisan KodeTailwind Optimization: Gunakan utility classes secara efisien. Jangan mengulang-ulang konfigurasi CSS yang sama jika bisa dijadikan komponen reusable.Modularitas: Pecah komponen menjadi bagian kecil agar saat ada error, AI hanya perlu mengirimkan ulang file kecil tersebut, bukan file besar (hemat token input/output).4. Perintah Spesial untuk Saga (User)Gunakan perintah "Brief mode on" jika kamu hanya butuh kode mentah tanpa penjelasan sama sekali.Gunakan perintah "Review plan only" untuk melihat progres tanpa meminta AI menulis ulang seluruh PRD.
